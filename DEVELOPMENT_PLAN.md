# 會計學習網站 — 開發計畫

> 本文件是本專案的**唯一需求來源（single source of truth）**，供執行開發的 AI 模型依循。
> 執行時逐 Phase 進行，每個 Phase 完成並通過驗收標準後 commit，再進入下一個 Phase。

---

## 1. 專案概述

**使用者背景**：會計初學者（本身是全端開發者，但會計領域從零開始）。

**目標（依優先序）**：
1. **主要**：學習可實際應用於台灣傳統產業的財務與稅務管理知識（帳務處理、營業稅／營所稅申報、扣繳、發票管理等）
2. **次要**：考取台灣「記帳士」證照（專技普考，考選部主辦，每年約 11 月舉行）

**產品定位**：單人使用的自學平台，內容深度貼合台灣現行法規，功能上是「教材 + 題庫 + 複習卡 + 實務工具」的整合網站。

**不設考試期限**：開發順序以「實務應用價值」優先，考試衝刺功能其次。

## 2. 需求範圍

### 2.1 核心功能（本計畫涵蓋）

| 功能 | 說明 |
|------|------|
| 教材／筆記系統 | 主題式教材，依科目→章→節組織，支援使用者附註筆記 |
| 題庫與測驗 | 選擇題練習、歷屆試題、錯題本、模擬考 |
| 間隔複習卡 | FSRS 演算法排程的 flashcards（法條、稅率、會計處理記憶點） |
| 實務練習工具 | 分錄練習器、營業稅／營所稅試算器 |
| 學習進度追蹤 | 儀表板：各科目完成度、答題正確率趨勢、複習卡到期數 |

### 2.2 明確不做（本期範圍外）

- 使用者註冊／登入（但資料層須預留擴充，見 §4.3）
- 網站內建即時 AI 功能（問答、即時生成題目）——所有內容於開發期預先生成
- 任何付費服務依賴——整體架構必須在免費額度內運行
- 原生 App——responsive web 即可（桌面優先、手機可用）

## 3. 技術棧決策

| 層面 | 選擇 | 理由 |
|------|------|------|
| 框架 | **Next.js（App Router）+ TypeScript** | 教材頁適合 SSG；生態成熟，後續模型好接手；未來要加後端（多人版）不用換框架 |
| 樣式 | **Tailwind CSS** | 快速開發、responsive 容易處理 |
| 內容格式 | **MDX**（教材）+ **JSON**（題庫／字卡／參數） | 內容即檔案，進 git 版控，AI 生成與人工校對都方便 |
| 本地儲存 | **IndexedDB（Dexie.js）** | 免後端儲存進度；容量遠大於 localStorage；透過抽象層隔離（§4.3） |
| SRS 演算法 | **ts-fsrs** | 現代 FSRS 演算法的成熟 TypeScript 實作 |
| 圖表 | **Recharts**（或同級輕量方案） | 儀表板進度圖 |
| 部署 | **Vercel 免費方案** | 零成本、自動部署、對 Next.js 零設定 |
| 測試 | **Vitest**（單元，僅核心邏輯：計分、SRS、試算器） | 遵守零過度工程，不做全面覆蓋 |

**成本結論**：全部落在免費額度內，無任何月費。

## 4. 系統架構

### 4.1 目錄結構

```
會計學習網站/
├── DEVELOPMENT_PLAN.md        # 本文件
├── CLAUDE.md                  # 執行模型的專案守則
├── content/                   # 所有學習內容（git 版控）
│   ├── materials/             # 教材 MDX，依科目分資料夾
│   │   ├── accounting/        # 會計學
│   │   ├── tax-practice/      # 租稅申報實務
│   │   ├── tax-law/           # 稅務相關法規
│   │   └── bookkeeping-law/   # 記帳相關法規
│   ├── questions/             # 題庫 JSON
│   │   ├── past-exams/        # 歷屆試題（依年度／科目）
│   │   └── generated/         # AI 生成練習題
│   ├── flashcards/            # 複習卡 JSON
│   └── tax-parameters/        # 年度稅務參數（見 §6.3）
│       └── 2026.json
├── src/
│   ├── app/                   # Next.js 路由
│   ├── components/
│   ├── lib/
│   │   ├── storage/           # 儲存抽象層（見 §4.3）
│   │   ├── srs/               # FSRS 包裝
│   │   ├── quiz/              # 計分、抽題邏輯
│   │   └── calculators/       # 稅額試算邏輯
│   └── types/                 # 內容 schema 的 TS 型別
└── scripts/                   # 內容匯入／驗證腳本
    ├── validate-content.ts    # schema 驗證 + 檢查 verified_at
    └── import-past-exam.ts    # 歷屆試題 PDF→JSON 輔助
```

### 4.2 資料 Schema

**教材 MDX frontmatter**：
```yaml
---
id: acc-01-03            # 科目縮寫-章-節
title: 借貸法則與會計恆等式
subject: accounting
chapter: 1
order: 3
law_basis: [商業會計法, 商業會計處理準則]   # 涉及的法規（可空）
verified_at: 2026-07-05   # 內容最後查證日期（必填，見 §6.2）
sources:                  # 查證來源 URL（官方來源優先）
  - https://law.moj.gov.tw/...
---
```

**題目 JSON**：
```jsonc
{
  "id": "pe-2025-tax-law-01",       // pe=歷屆 / gen=生成
  "subject": "tax-law",
  "source": { "type": "past-exam", "year": 2025, "number": 1 },  // 或 {"type":"generated"}
  "type": "single-choice",           // single-choice | multi-choice
  "stem": "題幹…",
  "options": ["…", "…", "…", "…"],
  "answer": [1],                     // 0-based index 陣列
  "explanation": "詳解…（生成題必填；歷屆題由 AI 補寫詳解）",
  "material_ref": "tax-01-02",       // 對應教材節，供錯題跳轉
  "tags": ["營業稅", "進項稅額"],
  "verified_at": "2026-07-05"
}
```

**複習卡 JSON**：
```jsonc
{
  "id": "fc-tax-0042",
  "subject": "tax-law",
  "front": "營利事業所得稅之課稅稅率為？",
  "back": "20%（課稅所得額超過 12 萬元部分；另有級距緩衝規定）",
  "material_ref": "tax-02-01",
  "verified_at": "2026-07-05"
}
```

**IndexedDB 使用者資料表**（Dexie）：
- `progress`：教材節閱讀狀態（materialId, status, updatedAt）
- `notes`：使用者對教材節的筆記（materialId, content, updatedAt）
- `attempts`：每次答題紀錄（questionId, correct, chosenAt）——錯題本與正確率由此推導
- `srs_cards`：每張卡的 FSRS 狀態（cardId, due, stability, difficulty, …）
- `exam_sessions`：模擬考紀錄（科目、題目清單、得分、用時）

### 4.3 多人擴充預留（只做隔離，不做實作）

所有使用者資料讀寫**必須**經過 `src/lib/storage/` 的單一介面（如 `StorageProvider`），元件不得直接 import Dexie。介面方法以 async 設計。未來要多人版時，實作一個 Supabase provider 替換即可，UI 層不動。**本期只做 IndexedDB 實作，不做帳號、不做同步。**

另提供「匯出／匯入 JSON 備份」功能（設定頁一顆按鈕），作為單機資料的保險，也是未來遷移到雲端的資料通道。

## 5. 內容架構（科目地圖）

依記帳士考科組織，並加一個考試以外的「實務專區」：

1. **會計學**（考科：會計學概要）——會計基本概念、分錄、過帳、試算、調整、結帳、財務報表、存貨、固定資產…
2. **租稅申報實務**（考科：租稅申報實務）——綜所稅、營所稅結算申報、營業稅（401/403 申報書）申報實務與計算
3. **稅務相關法規**（考科：稅務相關法規概要）——所得稅法、加值型及非加值型營業稅法、稅捐稽徵法
4. **記帳相關法規**（考科：記帳相關法規概要）——記帳士法、商業會計法、商業會計處理準則、公司法總則
5. **實務專區**（非考科，對應主要目標）——傳產公司實際帳務流程：發票開立與進項管理、每期營業稅申報流程、各類所得扣繳與申報、二代健保補充保費、勞健保費用處理、年度結算申報時程表

> **執行模型注意**：正式考科名稱與範圍以考選部（wcberereg.moex.gov.tw／moex.gov.tw）公告為準，開始生成內容前先以 WebSearch 查證當年度考科與命題大綱，若與上表不符，以官方公告修正上表。國文（作文）科不納入本網站範圍。

## 6. 內容生產流程（貫穿各 Phase 的持續工作）

### 6.1 AI 生成教材與題目

由 Claude Code 於開發期生成，流程固定為：

1. 依 §5 科目地圖擬定該科**章節大綱**，經使用者確認後才開始寫內容
2. 逐節生成教材 MDX：概念說明 → 台灣法規依據 → 實例（優先用傳產情境：製造業、買賣業）→ 常見錯誤
3. 每節配套生成 5–10 題練習題 + 5–10 張複習卡
4. 執行 §6.2 查證流程，填入 `verified_at` 與 `sources`
5. 跑 `scripts/validate-content.ts` 通過後 commit

### 6.2 法規時效驗證（強制，不可跳過）

台灣稅務數字每年變動（免稅額、扣除額、級距、起徵點等），**AI 記憶中的數字一律視為過時**。規則：

- 任何含具體數字（稅率、金額、門檻）或法條引用的內容，生成時**必須**以 WebSearch 對官方來源查證：全國法規資料庫（law.moj.gov.tw）、財政部稅務入口網（www.etax.nat.gov.tw）、國稅局網站
- 查證通過才能寫入 `verified_at`；查不到官方佐證的數字不得寫入內容，改以參數引用（§6.3）或標註「以當年度公告為準」
- `validate-content.ts` 檢查：所有內容檔必有 `verified_at`，且 CI 中對超過 12 個月未複驗的檔案發出警告清單

### 6.3 年度稅務參數檔

每年變動的數字集中在 `content/tax-parameters/<年度>.json`（免稅額、標準扣除額、營所稅率與級距、營業稅率、扣繳率、二代健保費率、基本生活費等）。教材 MDX 以自訂元件 `<TaxParam id="..."/>` 引用，試算器直接讀此檔。每年只需更新一個檔案，全站數字同步。

### 6.4 歷屆試題匯入

1. 從考選部「考畢試題查詢平臺」（wq-doc.moex.gov.tw 或 moex 官網）下載記帳士近 5 年、4 個專業科目的試題與公告答案 PDF
2. 以 `scripts/import-past-exam.ts` 輔助 + AI 逐題轉為題目 JSON（選擇題全收；申論／計算題收錄題幹與參考擬答方向，型別另訂 `essay`，僅供閱讀不計分）
3. 每題由 AI 補寫詳解，詳解中的法規數字同樣走 §6.2 查證（舊試題若因修法答案已不同，於詳解中標註「本題依當年法規，現行規定為…」）
4. 版權注意：考選部試題為政府公開資訊，收錄時標明出處年度與科目

## 7. 開發階段

> 每個 Phase 開一支 `feat/` 分支，完成驗收後 commit。Phase 內小任務也隨做隨 commit。

### Phase 0：專案基礎（半天）

- 建立 Next.js + TypeScript + Tailwind 專案，git init
- 設定 Vercel 部署（連 GitHub repo，push 即部署）
- 建立 §4.1 目錄骨架、§4.2 各 schema 的 TS 型別、`validate-content.ts` 雛形
- 建立 `StorageProvider` 介面與 Dexie 實作骨架

**驗收**：`npm run dev` 可跑；Vercel 上看得到首頁；`npm run validate-content` 可執行。

### Phase 1：教材系統 + 第一批內容

- 教材瀏覽 UI：科目列表 → 章節側欄 → MDX 內容頁（含 `<TaxParam>` 元件）
- 節末「標記已讀」與使用者筆記（存 IndexedDB）
- 全站搜尋（教材標題／標籤，客製 build-time 索引即可，不引入外部服務）
- **內容**：完成「會計學」第 1–3 章 + 建立 `tax-parameters/2026.json`（走 §6.2 查證）

**驗收**：可從首頁進到任一教材節閱讀；筆記重新整理後仍在；參數元件正確渲染；內容檔全數通過 validate。

### Phase 2：題庫與測驗 + 歷屆試題

- 練習模式：依科目／章／標籤抽題，即答即解；詳解旁提供「看不懂」回饋鈕（存 IndexedDB，供 §8.3 優化循環使用）
- 模擬考模式：依科目、計時、交卷後計分與逐題檢討
- 錯題本：自動收集答錯題，支援重練、掌握後移除
- 歷屆試題匯入：近 5 年 × 4 科（§6.4 流程）
- 每題「回教材」連結（material_ref）

**驗收**：完成一次模擬考流程無誤；錯題本邏輯正確（答對後可移除）；歷屆題庫至少 3 個年度上線且抽查 10 題與官方答案一致。

### Phase 3：實務練習工具（對應主要目標）

- **分錄練習器**：給交易情境（如「本公司開立三聯式發票銷貨 105,000 元含稅」），使用者選會計科目、填借貸金額，系統核對並詳解；題目為 JSON 內容檔，先做 30 題傳產常見情境
- **營業稅試算器**：輸入銷項／進項，計算應納（溢付）稅額，對照 401 申報書欄位說明
- **營所稅試算器**：輸入全年收入／成本費用，依現行稅率級距（讀 tax-parameters）計算，含擴大書審概念說明
- 實務專區教材（§5 第 5 項）第一批：營業稅申報流程、扣繳實務

**驗收**：分錄練習可完整作答與批改；兩個試算器與手算結果一致（寫 Vitest 案例鎖住）；試算器數字全部來自 tax-parameters 檔。

### Phase 4：間隔複習卡 + 儀表板

- FSRS 排程：每日到期卡片佇列、四鍵評分（忘記／困難／良好／簡單）
- 卡片瀏覽器：依科目瀏覽全部卡片、暫停個別卡片
- 手機版重點打磨：複習卡與練習模式在手機上單手可操作
- 儀表板首頁：各科教材完成度、近 30 天答題正確率、今日到期卡數、模擬考歷次分數
- 儀表板提供「學習分析報告」一鍵匯出（§8.3）
- 設定頁：JSON 備份匯出／匯入（§4.3）、啟用 `navigator.storage.persist()`、距上次備份超過 30 天時顯示提醒

**驗收**：卡片評分後 due 日期依 FSRS 變化正確；手機 375px 寬度下複習與刷題可正常操作；備份匯出再匯入後資料一致。

### Phase 5：內容補完（持續進行）

- 依 §5 地圖補完四個考科所有章節教材＋配套題目／卡片
- 實務專區補完（二代健保、勞健保、年度申報時程表等）
- 建立 `docs/maintenance/` 年度維運 checklist（§8.1 三個時間點各一份，含逐步操作指引）
- 每完成一科跑一次 validate 並 commit

**驗收**：科目地圖上所有章節皆有教材；每節皆有 ≥5 題與 ≥5 卡；維運 checklist 齊備，進入 §8 維運循環。

### 未來擴充（不在本計畫內，僅記錄方向）

- 多人版：Supabase Auth + Postgres，實作第二個 StorageProvider
- 網站內建 AI 問答助手（需 API key 與後端）
- 讀書計畫排程（若之後決定報考年度，以考試日倒推）

## 8. 長期維運與自我優化（五年持續執行機制）

本專案預計持續使用五年以上。Phase 0–5 完成後進入**維運循環**，以下機制確保內容不過時、網站依實際學習狀況持續進化。核心原則：**每項循環任務都要落成 repo 內的腳本＋文件化步驟，任何一代 AI 模型讀完本文件即可執行，不依賴任何對話記憶。**

### 8.1 年度更新循環（每年固定執行）

| 時間 | 任務 |
|------|------|
| 每年 1 月 | 建立新年度 `tax-parameters/<年>.json`（WebSearch 查證財政部新公告的免稅額、扣除額、稅率、費率）；跑 `validate-content.ts` 取得逾期內容清單並全數複驗（§8.2） |
| 每年 4–6 月 | 查核考選部公告的當年度記帳士考科與命題大綱，若有修訂則更新 §5 科目地圖並調整對應內容 |
| 每年 12 月 | 匯入當年度記帳士歷屆試題（§6.4 流程）；執行年度檢討（§8.4） |

各任務的操作步驟寫在 `docs/maintenance/`（Phase 5 建立），格式為可直接照做的 checklist。

### 8.2 內容時效管理

- `verified_at` 超過 12 個月的內容自動列入「待複驗清單」（`validate-content.ts` 輸出）
- 複驗＝重新走 §6.2 查證流程：法規未變則僅更新 `verified_at`；有修法則修正內容並於文末加「修法紀錄」（原規定 → 現行規定 → 生效日）
- 「待複驗清單清空」是每年 1 月循環的驗收標準

### 8.3 數據驅動的內容自我優化（每季執行）

儀表板提供「學習分析報告」一鍵匯出（Markdown），內含：各章節答題錯誤率排行、重複答錯 ≥3 次的題目、SRS 中反覆遺忘（lapse 數高）的卡片、超過 60 天未讀的章節、被標記「看不懂」的詳解。

使用者將報告交給 Claude Code，依報告執行優化：

1. 錯誤率最高的 3 個章節 → 改寫教材（補傳產實例、白話化）並增加練習題
2. 反覆遺忘的卡片 → 拆成更小的記憶單位、重寫記憶點
3. 被標記「看不懂」的詳解 → 重寫詳解
4. 久未讀章節 → 於檢討時判斷是內容問題還是排序問題

### 8.4 年度檢討與計畫演進

- **本文件是活文件**：需求、範圍、技術決策變更時直接修改本文件並 commit，同步在文末「變更紀錄」加一行（日期＋摘要＋理由）
- 每年 12 月檢討：實際使用頻率低的功能考慮精簡；缺口納入下年度；§7「未來擴充」清單是否有項目該啟動（例如決定報考年度後，啟動讀書計畫排程）
- 重大技術變更（換框架、加後端、引入付費服務）必須先在變更紀錄寫明理由與影響範圍，經使用者確認後才動工

### 8.5 技術可持續性

- 依賴選擇偏保守：只用主流且活躍維護的套件；每年 1 月循環時執行 `npm outdated` 檢視，小版本順手升級，major 升級僅在安全性需要或阻礙開發時進行並單獨開分支
- 資料保全（五年學習紀錄不可遺失）：啟用 `navigator.storage.persist()` 降低瀏覽器清除風險；距上次備份超過 30 天於儀表板顯示提醒；備份 JSON 含 `schemaVersion` 欄位，日後資料結構演進時提供遷移函式，確保多年前的舊備份永遠可匯入
- 部署平台若免費方案政策改變（Vercel 條款調整等），優先遷移至其他免費靜態託管（GitHub Pages、Cloudflare Pages），架構上不依賴 Vercel 專屬功能

## 9. 給執行模型的守則

1. **繁體中文**：所有 UI 文字、內容、commit message 描述皆用繁體中文；會計／稅務用語採台灣慣用語（如「營所稅」非「企業所得稅」）
2. **法規查證是硬性規定**：見 §6.2，含數字的內容未經官方來源查證不得進 repo
3. **零過度工程**：不加本計畫未列的功能與抽象；唯一的預留抽象是 `StorageProvider`
4. **內容與程式分離**：程式碼不得硬編碼任何稅務數字，一律讀 tax-parameters
5. **存檔點習慣**：每個小任務完成即 commit；Phase 完成開 PR（若無遠端 repo 則 merge 回 main）
6. **內容大綱先確認**：每科教材動筆前，先把章節大綱給使用者確認（遵守「新功能先 spec」原則）
7. **可交接性**：所有維運循環任務（§8）必須落成 repo 內的腳本與 `docs/maintenance/` 文件，執行結果 commit；不得依賴任何對話記憶或特定模型
8. **計畫同步**：任何偏離本計畫的決定，先更新本文件與文末變更紀錄，再動工
